## Control Context and Constraints

- Context: limited critical resources
  - Need to provide complete information: Fill the bucket
  - Prevent incorrect information: correct/reset errornous context ASAP
  - Use specific information --> inject updated data via MCP
  - Constrain to project/team culture

### Fill the bucket

1. Start with analyse current code base about the specific area we want to change in `ASK` mode
   - Then ask about options to achieve the requirements --> human review of options
2. Ask for a summary of important things with the "chosen" option
3. Switch to `PLAN` mode, in another session, to plan with the above summary
   - While review the plan, may ask for details of section that are not clear
   - May need to go back to [2] when some section wasn't completed/cleared
4. Perform the implementation in `AGENT` mode
5. Review & may switch back to `ASK` mode to understand the code
   - May also update related documentation

### Avoid context poisoning

- Poisoned context: bad output from vague initial prompt create errors in context --> incorporated & influenced later works
  - May resulted in unecessary defensive actions --> over-compensated action by AI
  - Better off to restart the session, or at least re-start at the point of error to remove the "poisoned context"
- Context scoping strategy: ask for a summary, which are reviewed and copied to new session

### Inject focused documentation

- Model depend on general, training data that may be out-of-date
  - Even with specific guiding prompt, AI can't do what it doesnt know (e.g. using features from newer version of a library)
  - Utilise MCP-server to feed more up-to-dated, or more focus documentation for the model

### Record project/team culture

- Can use AI to review code & identify repeated patterns --> project culture
  - Consolidated these into AI rules/workflows to be applied for future codes generated by AI itself

## Design Custom Agents/Modes

- Agent: instructions with access to tools --> unit of work for an AI-based solution
  - Each run in its own isolated session --> control context
  - Agents can be chained together --> pipeline
- Sample agents break down:
  - Plan: User stories, specification
  - Test: UnitTest, integration test
  - Code: EF entities, migration, services and endpoints, etc --> project & technology specific
- Agents should be specific for your team, project and culture --> will need to be updated overtime
  - To develop these agent, need ability to spot good solution --> know what to look for, and ability to compare solution
- Building agent:
  1. Start from a general prompt
  2. Run with sample case, review --> update the agent instruction
  3. Repeat step 2 until satisfied

## Orchestrate Workflow and Processes

- To orchestrate "correct" program, we need two main types of inputs:
  1. User Prompt: what user want --> iteratively toward the program goal
  2. Skills: define iteration constraints: how to iterate, specilized guidance to maintain quality in code generation
     - only load relevant parts of skills to avoid pollute the context
- Could create "orchestrator" agent, which can delegate its tasks to specialized agents
  - The cost of agent usage seems less --> doesn't cover the cost of creating & maintaining the various agents, nor the human in the loop
  - Using multi-shots approach --> demanding continuous human attention, but may resulted in higher quality
  - But after initial setup, coding loop maybe much shorter (note: *Could developer really read through & understand all the generated code ?* )
